# 🏦 Банковский счет: Транзакции, Исключения и История (25.02.2026)

[![C++](https://img.shields.io/badge/C++-17%2B-blue.svg)](https://isocpp.org/)

Привет! В этой директории расположен продвинутый класс `BankAccount`, который не просто хранит баланс, но и реализует строгую валидацию данных, безопасные переводы между счетами и ведет историю операций (с метками времени).

---

## 📋 Основные возможности (`BankAccount`)

В файле `BankAccount.cpp` (и соответствующем `.h` файле) реализован следующий функционал:

1. **Создание счетов:** Автоматическая генерация уникального номера счета (ACC4001, ACC4002 и т.д.) через `static` переменную.
2. **Финансовые операции:** 
   * `deposit` (Пополнение)
   * `withdraw` (Снятие)
   * `transfer` (Перевод на дру��ой счет)
3. **Безопасность и Валидация:** 
   * Строгие проверки на отрицательные значения, пустые имена и даже проверки на то, является ли число конечным (`std::isfinite`).
   * Защита от перевода самому себе (`this == &targetAccount`).
   * Обработка нехватки средств.
4. **История транзакций:** Каждая успешная операция фиксируется в массиве `history` вместе со временем (`std::time`).

## 🚀 Архитектурные фишки и Code Review

Код написан на очень хорошем уровне, в нем прослеживается понимание того, как работают реальные отказоустойчивые системы:

### 1. Транзакционный подход при переводах (Rollback)
Метод `transfer` — это жемчужина данного класса! 
```cpp
// списываем
balance -= amount;
try {
    // зачисляем
    targetAccount.balance += amount;
    // история ...
} catch (...) {
    // rollback (откат транзакции в случае ��боя)
    balance += amount;
    throw;
}
```
Это классический паттерн программирования безопасных систем. Если во время зачисления или записи в историю произойдет ошибка (например, исключение выделения памяти объектом строки), банк откатит списание, и деньги клиента не «сгорят» в пустоте.

### 2. Защита от NaN и Бесконечности
Функция `validateAmountFinite(double value)` проверяет `std::isfinite`. Это очень важная деталь при работе с `double` — она защищает банк от хакеров или багов, при которых на баланс могло бы прийти значение `Infinity` или `NaN`.

### 3. Перегрузка операторов вывода (`operator<<`)
Класс правильно делегирует вывод своих данных в потоки `std::ostream`. Мы можем легко написать `std::cout << account;` или `std::cout << operation;`, и получить красиво отформатированный вывод без вызова громоздких методов `print()`.

## ⚠️ Что можно улучшить (Точки роста)

1. **Ограниченная история (`historySize >= maxOperations`):**
   В методе `addToHistory` есть проверка: если история переполнена, функция просто возвращает `return;`. Это означает, что старые переполненные счета **перестанут записывать историю молча**. Лучше использовать динамический массив `std::vector<Operation> history`, который может расти автоматически.
2. **Перевод каретки `std::ctime`:**
   Функция `std::ctime(&operation.date)` возвращает строку со временем, которая **уже содержит** символ переноса строки (`\n`). Из-за этого при формировании логов могут возникать неожиданные визуальные разрывы.
3. **Обнуление счетчиков:**
   Статическая переменная генерации `id` всегда растет. В многопоточном приложении (`multithreading`) п��остой инкремент `id++` может привести к состоянию гонки (Race Condition). В будущем стоит использовать `std::atomic<int>`.
